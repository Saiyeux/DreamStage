# 代码评审与重构提示词清单

> 用于开发者自检、代码评审（Code Review）和安全重构（Refactoring）的标准化提示词。建议集成到 PR 模板或团队 Wiki 中。

---

## 🧾 代码评审提示词（Code Review Checklist）

### ✅ 功能性
- [ ] 代码是否正确实现了需求/用户故事？
- [ ] 是否处理了所有边界条件和异常情况？
- [ ] 是否有未处理的 `TODO`、`FIXME` 或临时调试代码？

### ✅ 可读性与命名
- [ ] 变量、函数、类名是否清晰、准确、符合业务语义？
- [ ] 逻辑是否易于理解？是否需要添加注释或拆分函数？
- [ ] 是否避免了“魔法数字”或“魔法字符串”？

### ✅ 设计与结构
- [ ] 是否遵循单一职责原则（SRP）？
- [ ] 是否存在重复代码？是否可以提取公共逻辑？
- [ ] 模块/类之间的耦合是否合理？是否便于测试和扩展？

### ✅ 测试
- [ ] 是否有对应的单元测试/集成测试？
- [ ] 测试是否覆盖主要路径、边界条件和错误场景？
- [ ] 测试是否具有可读性和可维护性（如使用 AAA 模式）？

### ✅ 性能与安全
- [ ] 是否存在明显的性能问题（如 N+1 查询、大循环内调用 API）？
- [ ] 是否对用户输入做了校验和转义？是否存在 XSS、SQL 注入等风险？
- [ ] 敏感信息（如密钥、密码）是否被硬编码？

### ✅ 代码风格与规范
- [ ] 是否符合团队的代码格式规范（缩进、空格、换行等）？
- [ ] 是否通过了 Linter / Formatter（如 ESLint、Prettier、Black、Checkstyle）？
- [ ] 是否使用了不推荐或已废弃的 API？

### ✅ 可维护性
- [ ] 日志是否合理？关键路径是否有 traceable 日志？
- [ ] 配置是否可外部化（如通过环境变量或配置文件）？
- [ ] 是否引入了不必要的依赖或过重的库？

## 🔧 重构提示词（Refactoring Guidance Prompts）

### 🔍 识别重构点
- 这段代码是否让新成员难以理解？
- 我是否在多个地方复制粘贴了相似逻辑？
- 修改一个小功能是否需要改动多个不相关的文件？
- 单元测试是否难以编写或运行缓慢？

### 🛠️ 重构策略
- **命名是否准确？** → 优先重命名（Rename），这是最安全且高回报的重构。
- **函数是否太长？** → 提取逻辑到小函数（Extract Function）。
- **类是否承担太多职责？** → 拆分类（Split Class）或提取策略。
- **条件逻辑复杂？** → 考虑多态、策略模式或卫语句（Guard Clauses）。
- **重复代码？** → 提取到工具函数、基类或共享模块。

### ✅ 安全重构原则
- [ ] **先写测试**：确保有足够测试覆盖再开始重构。
- [ ] **小步提交**：每次只做一种重构，频繁提交并运行测试。
- [ ] **不混合功能与重构**：重构应单独提交，避免与新功能混在一起。
- [ ] **使用 IDE 重构工具**：如 VS Code / IntelliJ 的“Extract Method”、“Rename Symbol”等，减少人为错误。

### 📈 重构后验证
- [ ] 所有现有测试是否仍然通过？
- [ ] 应用行为是否与重构前完全一致？
- [ ] 代码复杂度（如圈复杂度）是否降低？
- [ ] 新代码是否更容易添加新功能或修复 Bug？

## 💡 团队协作建议（可加入 PR 模板）

> **请评审人关注：**
> - 本次变更是否引入技术债务？
> - 是否有更好的设计或更简洁的实现方式？
> - 是否有潜在的并发、性能或安全问题？

> **开发者自述（建议填写）：**
> - 本次改动最难的部分是什么？
> - 哪些地方你不太确定，希望获得反馈？
> - 是否已本地验证核心流程？
